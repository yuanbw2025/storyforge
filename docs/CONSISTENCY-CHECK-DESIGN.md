# 一致性检测 与 多故事线进度追踪 · 开发设计文档

> 对应 ROADMAP：Phase 38（AI 生成内容一致性检测）、Phase 39（主角多故事线进度追踪）
> 状态：设计稿（待实现）｜创建：2026-06-04
> 原则遵循：一切文本分析与内容提取均调用 AI 实现，不使用正则。

---

## 0. 背景与目标

实测各模型在长篇二次承接时的幻觉率（按情节计，非字数比例）：豆包约 10%、DeepSeek 近 30%，且常违背指令、篡改关键情节；提示词最多压到约 5%。结论是 **AI 生成后仍须人工核查**。

本设计的目标，是把"人工逐字核查"升级为 **"AI 先标红、人来定夺"**：

- **Phase 38**：以工具已沉淀的事实库为基准真相，对新生成正文做矛盾检测，定位"与前文逻辑不符"的点。
- **Phase 39**：解决"主角同时挂多条故事线时，如何追踪各线进度、识别交叉"的难题，并把线索状态反哺给 Phase 38。

两者关系：Phase 39 产出的"活跃/已收束线索 + 交叉节点"是 Phase 38 事实基准的一部分；Phase 38 可独立先行（MVP 不依赖 39）。

---

## 1. 设计原则

1. **事实库即基准真相**：检测不凭空判断对错，而是以"已确立事实"为尺子量"新内容"。
2. **闭集分类，而非开放抽取**：凡需 AI"归类/对照"的环节，都先给定已知集合（角色清单、线索注册表），让 AI 映射其上，绝不每次凭空发明——这是规避 AI 漂移、重复、幻觉的关键手法（与角色去重修复同源）。
3. **手动触发，节约 token**：检测与进度追踪均由用户按需发起，不做生成后自动触发。作者自信可自行审校时即可省去开销。
4. **作者在环**：AI 只"提议 + 定位 + 映射"，关键决策（采纳冲突、确认新线索、改名合并）始终交作者。
5. **不造假精度**：进度用粗粒度阶段 + 一句话描述，不强行输出百分比。
6. **复用优先**：最大化复用既有的状态表、物品栏、年表、关系、审校链路，避免重复造轮子。

## 3. Phase 38 — AI 生成内容一致性检测

### 3.1 与既有"章节审校"的边界

- **章节审校（review-adapter，Phase F1）**：编辑视角，输出"质量评分 + 泛化建议"，五维（逻辑/人物/世界观/伏笔/节奏），上下文只塞少量碎片，且**未接入物品栏/年表/关系**。
- **一致性检测（本功能）**：事实视角，输出"冲突点列表 + 撞上的既定事实 + 处置建议"，是"幻觉定位"，非"质量打分"。

二者**并存、互补、不合并**。

### 3.2 数据流

```
[用户在编辑器点"一致性检测"]
        │
        ▼
①事实基准上下文组装器  ← 状态表 + 物品栏 + 故事年表 + 人物关系 + 力量层次 (+Phase39线索)
        │  （用 buildSelectiveStateContext 按需召回，控 token）
        ▼
②矛盾检测适配器(AI)    ← 输入：新正文 + 事实基准
        │  输出：冲突点列表（引文 ⚔ 既定事实 + 严重度 + 依据 + 建议）
        ▼
③预警对比 UI           → 每条冲突：忽略 / 采纳(回填事实库) / 跳转正文修改
```

### 3.3 待开发模块

**① 事实基准上下文组装器** —— 新建 `lib/ai/consistency/fact-context-builder.ts`

- 职责：把分散的事实源整合为一份结构化 ground truth 文本。
- 输入：projectId、当前章节正文（用于按需召回相关实体）、可选范围（聚焦主角/某些角色）。
- 整合内容：
  - 角色状态卡（位置、实力/境界、持有物、目标、关系等字段）
  - 物品栏当前持有（`aggregateInventory` 结果）
  - 故事年表近期关键事件（按 importance 与最近章节截取）
  - 人物关系网（与在场角色相关的关系）
  - 力量层次（现阶段取状态卡"实力/境界/等级"字段；Phase 34 后取阶段序号）
  - （Phase 39 就绪后）活跃/已收束线索 + 交叉节点
- 控 token：复用 `buildSelectiveStateContext` 的相关性召回，仅注入与本章相关的实体；分区块、各区设上限。

**② 矛盾检测适配器** —— 新建 `lib/ai/adapters/consistency-check-adapter.ts`

- `buildConsistencyCheckPrompt(factContext, chapterTitle, chapterText)` → `ChatMessage[]`
- `parseConsistencyResult(raw)` → `ConsistencyResult`
- 输出结构（建议）：
  ```ts
  interface ConsistencyConflict {
    severity: 'info' | 'warning' | 'critical'
    category: 'location' | 'item' | 'power' | 'relation' | 'timeline' | 'plot' | 'character' | 'other'
    quote: string          // 新内容中的冲突引文
    conflictsWith: string  // 撞上的既定事实（来自基准）
    reason: string         // 为何判定冲突
    suggestion: string     // 处置建议
  }
  interface ConsistencyResult {
    conflicts: ConsistencyConflict[]
    checkedEntities: string[]   // 本次纳入对照的实体，便于 UI 说明覆盖范围
  }
  ```
- Prompt 要点：明确"只报与给定事实冲突的点，不做文笔/质量评价；无冲突返回空数组；每条须引用新内容原文并指明撞了哪条事实"。

**③ 预警对比 UI** —— 新建 `components/editor/ConsistencyCheckPanel.tsx`

- 触发：编辑器工具栏新增"一致性检测"按钮（与"质量审校""提取状态"并列）。
- 展示：按 severity 排序的冲突卡片，左"新内容引文"、右"既定事实"对照。
- 每条操作：
  - **忽略**：标记为可接受（本次不处理）。
  - **采纳**：承认为合理剧情推进——把该变更并入事实库（与现有状态 diff 审核流打通，写状态卡/物品栏/关系）。
  - **跳转修改**：定位到正文对应位置由作者改写。

### 3.4 MVP 范围（建议先交付）

模块 ①②③ + 手动按钮，直接吃现有五套事实数据，**不改任何数据结构**。Phase 39 线索维度与 Phase 34 力量阶段作为后续增强接入。大纲级评估（3.6）、改进闭环（3.7）作为 Phase 38 的姊妹能力，可与 MVP 并行或紧随其后。

## 3-bis. 竞品借鉴与吸取（2026-06-04）

> 参考另一款工具的「逻辑评估——AI 检测大纲逻辑一致性与连贯性」功能，逐条对比后确定吸取项与取舍。本节是 Phase 38 维度体系、写前层、改进闭环的来源依据，**实现时须遵循其取舍判断，勿盲目照搬**。

### 3-bis.1 竞品做法（观察）

- **评估对象是大纲（章节级），不是正文**——在动笔前逐章查逻辑。
- 打分细化为 **3 组 12 维 + 进度条 + 数字**：逻辑组（因果逻辑链/时间线/世界观/逻辑连贯性）、角色组（角色行为/角色弧线/**战力**）、叙事组（伏笔/节奏/信息密度/悬念钩子）。
- **结构位置标签**（中段/结尾等），按章节在全书位置调整预期。
- 改进建议具体到缺失项（"是什么行为触发了追杀？主角拿到了什么数据？"）。
- **检测→行动闭环**：单章改进 + 批量改进（列表批选多章一键改）。
- 顶部三标签：评估 / 对比 / 历史（保留历史、改前改后对比）。

### 3-bis.2 吸取项（纳入本设计）

1. **大纲级评估（写前层）** —— 我们完全缺失，且大纲短、省 token，能在动笔前拦逻辑硬伤。与 Phase 38 互补：**大纲评估=写前预防，Phase 38 事实检测=写后深查**。详见 3.6。
2. **检测→改进闭环（单章/批量改进）** —— 当前我们"只标红不动手"是最大体验差距。补"一键按建议改进该章节点 / 批量改进选中节点"。详见 3.7。
3. **分组可视化维度** —— 输出重构为"逻辑组/角色组/叙事组"进度条卡片，并补 **战力 / 角色弧线 / 信息密度 / 悬念钩子**（战力即用户关注的力量层次）。详见 3.5。
4. **结构位置感知** —— 评估时告知 AI 该章处于开篇/中段/高潮/结尾，节奏与悬念预期不同。详见 3.5。
5. **历史 / 对比** —— 保留评估历史，支持改前改后对比（后续增强）。

### 3-bis.3 取舍与坚持（**实现时务必遵守，否则跑偏**）

1. **数字分评只做"排序提示"，不当结论**。AI 给每维打 0-100 看着专业，但单维数字噪声大、跨次会飘。**真正有价值的是锚定原文的具体问题（quote + 依据）**，我们现有的 `severity + quote` 比纯数字更扎实，必须保留为一等公民；分数仅用于排序/概览。
2. **不丢"事实库基准"这条护城河**。竞品像是逐章孤立自评，大概率发现不了"第 8 章得令牌、第 30 章令牌却还在原主手里"这类**跨章事实矛盾**。Phase 38 对照状态/物品/关系/力量/线索的深查能力，是我们强于竞品之处，**不可被大纲级自评取代**，二者并存。
3. **两层定位写死，别混淆**：
   - **大纲级逻辑评估（写前）** = 自洽 + 连贯 + 节奏，便宜、预防，对象是大纲节点。
   - **Phase 38 事实一致性检测（写后）** = 与事实库的矛盾定位，深查，对象是正文。

## 3.5 维度体系（分组，写前/写后共用）

将评估维度统一为 3 组，UI 以进度条 + 数字呈现，每维**尽量附锚定原文/大纲的具体问题**（无问题则只给分）。

- **逻辑组**：因果逻辑链、时间线、世界观一致性、逻辑连贯性
- **角色组**：角色行为一致性、角色弧线、战力（力量层次；Phase 34 阶段化后更精确）
- **叙事组**：伏笔衔接、节奏、信息密度、悬念钩子

**结构位置感知**：评估每个章节/节点时，传入其结构位置（开篇 / 中段 / 高潮 / 结尾，可由大纲卷章序推断），AI 据此调整节奏与悬念的合理预期。

> 注：12 维仅为初始集合，可按题材精简。分数为排序用途，问题锚定为核心产出（见 3-bis.3）。

## 3.6 大纲级逻辑评估（写前层）

- **对象**：大纲节点（卷/章），而非正文。复用 3.5 维度体系。
- **新建适配器**：`adapters/outline-logic-eval-adapter.ts`
  - 输入：大纲节点序列（标题 + 摘要 + 卷章结构）+ 世界观/故事核心/已建事实（轻量）。
  - 输出：逐节点 `{ 分组维度评分, 结构位置标签, 锚定问题列表, 改进建议 }`。
  - 因大纲短，可一次评估整卷/全书多节点（注意 token 上限，必要时分批）。
- **UI**：在大纲面板挂"逻辑评估"侧栏（参照截图布局）：逐章评分卡 + 进度条 + 位置标签 + 改进建议 + 勾选框；底部"开始评估""批量改进 (N)"。

## 3.7 检测→改进闭环（单章 / 批量改进）

- **单章/单节点改进**：对某条章节/大纲节点，按 AI 改进建议生成修订稿，作者预览后采纳（覆盖该节点摘要或正文）。
- **批量改进**：列表批选被标红的多个节点 → 串行逐个生成改进 → 汇总预览 → 作者逐条/一键采纳。
- **新建适配器**：`adapters/improve-from-eval-adapter.ts`（输入：原节点 + 该节点评估问题 → 输出：改进后的内容）。
- **作者在环**：改进稿一律先预览后采纳，绝不静默覆盖。
- **token 提示**：批量改进会显著消耗 token，UI 须显示选中数量并二次确认（与"手动触发省 token"原则一致）。

## 3.8 历史与对比（后续增强）

- 保存每次评估快照（评分 + 问题），支持"对比"视图查看改前/改后变化与分数走势。
- 优先级低于 3.5–3.7，作为 Phase 38 成熟后的体验增强。

## 4. Phase 39 — 主角多故事线进度追踪

### 4.1 难点

一个主角同时挂多条故事：必有一条主线 + 多条支线。需识别"主角在每条线上的当前进度"，并处理"多线交叉"。开放式逐章识别极易漂移、重名、错配。

### 4.2 核心思路：闭集分类 + 复用 StoryArc

把"开放识别"重构为"对已知线索集合的分类"：

1. **`StoryArc`（Phase B）作为"线索注册表"**：它已是作者事前规划的静态蓝图（主线/支线 + 起承转合阶段 `StoryStage[]`）。**不另造重复概念。**
2. **新增动态追踪层 `StorylineProgress`**：把已写正文映射回这些线，维护每线"当前进度指针 + 活跃状态 + 交叉节点"。

### 4.3 数据模型（新增）

```ts
interface StorylineProgress {
  id?: number
  projectId: number
  arcId: number               // 指向 StoryArc（线索注册表）
  currentStageId?: string     // 当前所处阶段（指向 StoryArc 的某个 StoryStage）
  status: 'dormant' | 'active' | 'climax' | 'resolved' | 'abandoned'  // 蛰伏/进行中/高潮/已收束/弃坑
  progressNote: string        // 一句话当前进度（最近状态，自由文本）
  lastActiveChapterId?: number
  involvedEntities: string    // JSON：相关角色/物品/地点/势力（交叉检测用）
  updatedAt: number
}

interface StorylineCrossing {        // 交叉节点
  id?: number
  projectId: number
  arcIdA: number
  arcIdB: number
  chapterId: number
  note: string                       // 交叉处发生了什么、产生的关联事实
  createdAt: number
}
```

### 4.4 每章归属与推进（AI，手动触发）

复用 story-timeline 抽取链路，新建 `adapters/storyline-track-adapter.ts`：

- 输入：**线索注册表**（各线 id + 名称 + 目标 + 当前阶段 + 相关实体）+ 新章节内容。
- 输出：
  - 本章推进了哪条/哪几条线（**对已知集合分类**）
  - 各线发生了什么、推进到哪个阶段、新的一句话进度
  - **新线索候选**（标记待作者确认，防漂移）
  - **交叉**（同一章推进 ≥2 条线，或两线共享实体本章互相影响）

### 4.5 交叉作为数据，而非要规避的问题

交叉记为 `StorylineCrossing` 链接节点。交叉是有价值的结构——如"此处交叉后林月已知秘宝之事，后文她若再表现不知情即为矛盾"——可直接作为 Phase 38 的事实喂入检测。

### 4.6 进度表达（不造假精度）

粗阶段指针（起/承/转/合）+ 一句话进度 + 最近活跃章，渲染为可读仪表盘：

```
复仇主线   【转·高潮前】 已查明仇人是城主，正集结力量      （最近活跃：第48章）
感情线·林月 【承】       互生情愫但因身份未挑明              （第45章）
秘宝支线   【合·已收束】 秘宝已得，线索关闭                  （第40章收束）
```

### 4.7 作者在环（防漂移关键）

线索注册表由作者确认/编辑；AI 只"提议候选 + 映射进度"，绝不每章自由发明线名。新线索候选走"提议 → 作者确认/改名/合并/忽略"流程。

### 4.8 待开发清单

1. `StorylineProgress` + `StorylineCrossing` 类型、DB 表（DB 版本 +1）、store。
2. `adapters/storyline-track-adapter.ts`：归属 + 推进 + 交叉/新线检测。
3. 线索进度仪表盘 UI + 交叉节点可视化。
4. 与 Phase 38 打通：把"活跃/已收束线索 + 交叉节点"纳入事实基准（检测"已收束线被错误重启""蛰伏支线角色突然按活跃线行动"等）。
5. 新线索候选作者确认流。

### 4.9 边界

- `StoryArc` = 静态规划蓝图（作者写）；`StorylineProgress` = 动态进度追踪（AI 从正文回填）。
- `story-timeline` = 扁平事件流；本功能 = 把事件归类到线索并维护每线进度指针。

## 6. 实现顺序建议

1. **Phase 38 MVP（写后事实检测）**：事实基准上下文组装器 → 矛盾检测适配器 → 预警 UI → 手动按钮。（不动数据结构，最快交付护城河能力）
2. **Phase 38 姊妹层（写前 + 闭环，吸取竞品）**：
   - a. 维度体系（3.5，分组 + 战力/角色弧线/信息密度/悬念钩子 + 位置感知）——写前/写后共用。
   - b. 大纲级逻辑评估（3.6）`outline-logic-eval-adapter` + 大纲侧栏 UI。
   - c. 检测→改进闭环（3.7）`improve-from-eval-adapter` + 单章/批量改进（含 token 二次确认）。
3. **Phase 39**：StorylineProgress 数据层 → 追踪适配器 → 仪表盘 UI → 反哺 Phase 38。
4. **历史/对比（3.8）**：评估快照 + 对比视图。（体验增强，低优先级）
5. **Phase 34**（可选）：力量阶段化，提升战力/力量维度检测精度。

> 关键取舍贯穿全程（见 3-bis.3）：分数只做排序提示、问题锚定原文为核心产出；事实库基准（写后深查）不可被大纲级自评取代，二者并存。

每步均：`npx tsc --noEmit` 零错误 + `npm run build` 成功 + 本地 `npm run dev` 走查。
